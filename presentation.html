01

The tension that shapes everything

The starter schema and the brief quietly contradict each other. Resolving that contradiction is the design.

Starter schema says confirmation_required: true

Ask the user before every order.

vs
The brief demands "…consistent decisions without input on each run."

Don't nag. Act.

Resolution

Confirmation stops being a blunt on/off switch and becomes a spending threshold the user dials up as trust grows. The default — auto_approve_under: 0 — confirms everything; raise it to grant autonomy. One field turns "safe" and "no-nag" from enemies into the same setting.

02

One decision, three verdicts

Every run resolves to exactly one verdict — no ad-hoc branches. The same engine handles normal orders and every failure.

AUTO
Act silently.
  • ≤ auto-approve threshold
  • within rolling cap
  • verified safe
CONFIRM
Ask first.
  • cost in the confirm band
  • rolling cap exceeded
  • falling back to backup
BLOCK
Refuse & notify.
  • over the hard ceiling
  • no compliant / safe option
  • never relax a restriction
The line that matters

Confirm is for what's within the user's authority but wants a yes. Block is for crossing a hard line — over-budget or unsafe. You never ask a human to approve the unsafe or the over-ceiling.

Uncertainty is never a yes.
Standing authority — the rule for silence & doubt

04

Consequence asymmetry

Preferences and restrictions get completely different machinery — because the cost of a miss differs by orders of magnitude.

Soft · preferences

Optimized for.

Used only to rank options. A miss is fine — you get a slightly less favorite meal.

cuisines · favorite_restaurants · dislikes

Worst case: mild annoyance.

Hard · restrictions

Absolute.

Used to filter + gate. Never violated; if it can't be verified, the candidate is dropped.

dietary · allergies · never_order

Worst case: the emergency room.

05

The decisions behind it

Six choices carry the design. The full 19-row rationale lives in references/trust-model.md.

01

Tiered autonomy, not a boolean

AUTO / CONFIRM / BLOCK with auto-approve as a trust dial — safe by default, never nagging.

02

Consequence asymmetry

Soft preferences rank; hard restrictions gate. Annoyance and the ER don't share a code path.

03

Deterministic safety

Allergy / dietary / budget are hard-coded — so it runs with any LLM, or none, and can't hallucinate a safety call.

04

Idempotency & reconciliation

One order per day-slot across manual, scheduled, retry, and crash — only a placed order burns the day. Plus a real-total budget re-check before the stop.

05

Self-throttle after a breach

The recovery model: a breach should cost the agent its own autonomy — the trust dial runs both ways. (Designed; ships with the spend ledger.)

06

Block ≠ Confirm

Confirm what's within authority; block what crosses a hard line. Never put the unsafe up for approval.

The model only ranks and phrases — so it can never hallucinate a safety decision.
Deterministic safety, optional intelligence

07

When prevention fails: earning trust back

The only post-order failure path. v1 ships the prevention side; this is the designed recovery playbook for when prevention fails — the post-order hooks, by inherited severity.

01

Contain harm first

For a P0, the first action isn't a refund — it's a loud "DO NOT EAT." Stopping someone eating an allergen outranks recovering $14.

02

Undo if possible

Attempt to cancel within the provider's window.

03

Own it — before the refund

Plain admission first: "I ordered X, which violates your restriction. My mistake." A silent refund isn't owning it.

04

Self-throttle

Drop auto-approve toward 0 and say so. Asking for less power is the apology — severity-scaled.

05

Learn

Log the episode; blocklist the item; distrust the bad data source — so it can't recur.

Trust is a dial: earned up, spent down.
Asking for less power is the apology

09

What's shipped — and what's next

Depth over breadth: prove the trust boundary first, ship the dangerous parts deterministic and tested — and the real provider behind them.

Shipped
  • Live DoorDash adapter — Playwright over a warmed profile: real menu, real cart, hard-stop before pay.
  • Deterministic engine — AUTO / CONFIRM / BLOCK, zero LLM calls; safety & budget in code.
  • Graceful degradation — favorite closed → next store, reported honestly; one-run-at-a-time profile lock; fail-closed carts.
  • Daily schedule + idempotency — OpenClaw cron claims one slot a day across manual / scheduled / retry / crash.
  • 91 tests — engine, config, run pipeline, adapter helpers.
Build next
  • Persisted spend ledger — enforce rolling caps & self-throttle across runs (today: per-run).
  • Any-store carting — handle every customization-modal layout, not just the common ones.
  • Schedule sync — drive the cron expression from order_time.
  • Dietary-uncertainty opt-in — confirm instead of drop.
  • Multi-meal slots — same claim / idempotency model.

One thing is deliberately never built: completing a charge. The adapter stops at the pay screen by design — a real charge would need an explicit flag, a typed env confirmation, and wiring this build omits. Stopping before pay is the feature, not a gap.

Think exhaustively about trust; ship the dangerous parts deterministic and tested.
